1. UPX 做了什么

UPX 对 exe 做了三件事：

原始 exe:
┌────────────────────────────────────┐
│ IMAGE_DOS_HEADER                   │
│ IMAGE_NT_HEADERS                   │
│ Section Table                      │
├────────────────────────────────────┤
│ .text  (原始代码)      ← 明文       │
│ .rdata (包含导入表)    ← 明文       │
│ .data  (全局变量)      ← 明文       │
│ .rsrc  (资源)          ← 明文       │
└────────────────────────────────────┘
           ↓ UPX 加壳后
┌────────────────────────────────────┐
│ UPX0 (未初始化数据)                │
│ UPX1 (压缩后的代码+数据+导入表信息)  │
│ UPX2 (解压 stub)    ← 唯一可执行代码│
└────────────────────────────────────┘

运行时，UPX stub 做两件事：

  1. 解压代码——把压缩的原始代码解压回内存
  2. 重建 IAT——调用 LoadLibrary + GetProcAddress，把函数地址填入 IAT 表
关键问题
UPX 在内存中重建的 IAT 是裸地址数组，不是 Windows 加载器认的 IMAGE_IMPORT_DESCRIPTOR 结构。所以直接 dump 内存得到的 exe 跑不起来。

2. 纯手动脱壳流程

2.1 寻找 OEP

UPX 的入口通常是这样的：

00401000 > pushad                    ; 保存所有寄存器（ESP 定律的关键）
00401001  mov esi, eip
...
00401050  popad                     ; 恢复寄存器
00401051  jmp 004F12E0              ; ← 跨段跳转到 OEP

ESP 定律法：

1. 在入口处，ESP = 0012FFC0（记下这个值）
2. 设置硬件访问断点：hr 0012FFC0
3. F9 运行
4. 断在 popad 处
5. F8 单步 → 看到 jmp OEP
6. F8 进入 OEP

单步跟踪法：

1. F8 单步走
2. 观察是否出现了跨段跳转（jmp .text 段）
3. 出现 → 跟上 → 到达 OEP

2.2 内存转储

到达 OEP 后：

用 x64dbg 的 Scylla 插件 → 点击 Dump
或者用 LordPE → 在目标进程右键 → 完整转储
→ 保存为 dumped.exe
验证 dump 是否正确
用 IDA Pro 或 x64dbg 打开 dumped.exe：
- 如果能反汇编出正常的代码 → dump 对了
- 如果全是乱码 → 还没到 OEP 就 dump 了

2.3 IAT 重建——体力活

dumped.exe 有原始代码但没有导入表，需要手动重建。

第一步：定位 IAT 地址

在 OEP 处的内存中，找到一段连续的指针值区域：

00402000  77E20000  ← 这是 MessageBoxA 的地址
00402004  77E10000  ← LoadIconA
00402008  00000000  ← 结束标志

这段区域就是 IAT，记下它的起始 RVA 和大小。

第二步：反查每个地址的函数名

对 IAT 中的每个地址，要找出它属于哪个 DLL、叫什么名字。这个过程在 Windows 中可以通过遍历进程已加载模块的导出表来手动反查，非常繁琐。

第三步：构建导入表结构

// 需要在 dump 文件的空白区写入以下内容：
//
// 1. DLL 名字符串（如 "USER32.dll\0"、"KERNEL32.dll\0"）
// 2. IMAGE_IMPORT_BY_NAME 结构（Hint + 函数名）
// 3. IMAGE_THUNK_DATA 数组（INT 和 IAT）
// 4. IMAGE_IMPORT_DESCRIPTOR 数组
// 5. 修改 PE 头 DataDirectory[1] 指向第 4 步的地址

这是纯手动脱壳中最耗时、最容易出错的一步。

纯手动 IAT 重建的工作量
假设一个 exe 导入了 3 个 DLL（USER32、KERNEL32、GDI32），共 50 个函数：
- 需要反查 50 个函数地址
- 写入 50 个 IMAGE_IMPORT_BY_NAME 结构
- 写入 50 个 IMAGE_THUNK_DATA（INT）+ 50 个（IAT）
- 写入 3 个 IID + 1 个结束标识
- 写入 3 个 DLL 名字符串
- 修改 PE 头的 3 处偏移
- 重新计算校验和

一个字节写错，整个 exe 就废了。

3. 哪些是体力活

步骤类型原因
找 OEP脑力活 ✅需要理解 UPX stub 的代码逻辑、知道 ESP 定律的原理、能识别 popad + jmp 跳转模式
内存 dump机械操作 ⚙️点一下按钮的事
定位 IAT 地址脑力活 ✅需要识别内存中的 IAT 区域
反查函数名体力活 💪遍历导出表、逐函数对比地址——机器最适合干这个
构建导入表结构体力活 💪机械地拼结构、算偏移、写字节——没有需要人类判断的地方
修复 PE 头体力活 💪算算偏移、改几个 DWORD
重要认知
"手动脱壳"在逆向圈的真实的含义是：手动找到 OEP + 手动 dump，IAT 修复交给工具。

重键 IAT 是纯粹的体力活——不需要理解算法、不需要做决策、不需要处理异常。就是机械地把 A 数据转换成 B 格式。这类工作恰恰是计算机最擅长的，没必要人去干。

4. 市面常用脱壳工具

工具用途特点
upx -dUPX 专用脱壳命令行，一键完成。只对标准 UPX 有效，修改过的 UPX 壳无法识别
Scylla通用脱壳（IAT 重建）x64dbg 内置插件，支持 x86/x64，可以脱各种壳
ImportREC (Import Recovery)通用脱壳（IAT 重建）老牌工具，只支持 32 位，功能稳定
LordPEPE 编辑 + 内存 dump查看/修改 PE 结构、dump 进程内存
OllyDump内存 dumpOllyDbg 的插件
UnPacKB自动脱壳机针对常见壳自动识别并脱壳

5. 实战：Scylla 一键修复

以 x64dbg + Scylla 为例，完整脱一个 UPX。以下是大致的操作流程。

环境准备

x64dbg（下载地址：https://x64dbg.com/）
自带 Scylla 插件

操作步骤

第一步：找到 OEP

1. 用 x64dbg 加载 upx.exe
2. 在入口处停下，看到第一行是 pushad
3. 记下 ESP 值（寄存器窗口能看到）
4. 右键 ESP → 在内存中跟随 → 选择硬件访问 → 字
5. F9 运行
6. 断下后 F8 单步 → 看到 "jmp XXX" 跨段指令
7. F8 跟上 → 到达 OEP

第二步：启动 Scylla

1. 在 OEP 处停下
2. 菜单：插件 → Scylla
3. 自动填充了 Imagebase 和进程 PID
4. OEP 栏填入当前 OEP 的 RVA

第三步：Dump

1. 点击 "Dump" 按钮
2. 选择保存路径 → 保存为 dumped.exe

第四步：自动修复 IAT

1. 点击 "IAT Autosearch" → 自动检测 IAT 区域
2. 点击 "Get Imports" → 自动扫描所有导入函数
   绿色 = 正常识别、红色 = 无法识别
3. 如果有红色的，点击 "Trace Level1" 或 "Level2"
4. 点击 "Fix Dump" → 选择 dumped.exe
5. Scylla 生成 dumped_SCY.exe

第五步：验证

1. 用 x64dbg 或 IDA 打开 dumped_SCY.exe
2. 检查字符串窗口 → 能看到原始字符串
3. 检查导入表 → 能看到 USER32.dll、KERNEL32.dll 等
4. 尝试运行 → 功能正常
Scylla 做了什么
Scylla 在后台做了我们第 3 节说的所有体力活：
1. 遍历 IAT 区域每个地址
2. 对每个地址，调用 GetModuleHandleEx 反查属于哪个 DLL
3. 遍历该 DLL 的导出表，反向查找函数名
4. 在 dumped.exe 的新增区块中写入完整的导入表结构
5. 修改 PE 头的数据目录表
整个过程不到 1 秒。

6. 总结

手动完成工具完成
理解 UPX 原理✅ 必须自己学—
找 OEP（ESP 定律）✅ 必须自己操作⚠️ 部分工具支持自动 OEP 搜索
验证 OEP 是否正确✅ 必须自己判断—
内存 dump❌ 没必要手动✅ LordPE / x64dbg
反查函数名❌ 没必要手动✅ Scylla / ImportREC
构建导入表结构❌ 没必要手动✅ Scylla / ImportREC
修改 PE 头❌ 没必要手动✅ 工具自动完成

记住一点：手动脱壳的"手动"是指手动找到 OEP。IAT 重建是体力活——你理解它的原理，但不要让自已去做它。
