1. PE 概述

PE（Portable Executable）是 Windows 平台的可执行文件格式，用于 EXE、DLL、SYS（驱动）、OCX、CPL 等文件。它基于 COFF 格式发展而来，保留了 DOS 兼容性。

PE 文件类型：

扩展名类型说明
.exeIMAGE_FILE_EXECUTABLE_IMAGE可执行文件
.dllIMAGE_FILE_DLL动态链接库
.sysIMAGE_FILE_EXECUTABLE_IMAGE驱动程序（内核模式）
.efiIMAGE_FILE_EXECUTABLE_IMAGEUEFI 固件程序

3. DOS Header 与 DOS Stub

DOS Header 是 PE 文件的历史遗留，前 2 字节固定为 MZ（Mark Zbikowski，DOS 开发者的名字缩写）。

typedef struct _IMAGE_DOS_HEADER {
    WORD  e_magic;     // "MZ" = 0x5A4D
    // ... 保留字段 ...
    LONG  e_lfanew;    // PE Signature 的文件偏移（最重要的字段！）
} IMAGE_DOS_HEADER;

e_lfanew（位于偏移 0x3C）是整个 DOS Header 中唯一还被使用的字段，它指向 PE Signature 的位置。

DOS Stub 紧跟在 DOS Header 后面，默认会输出 "This program cannot be run in DOS mode."。它的大小不固定，实际分析中可以忽略。

4. PE Signature 与 COFF Header

PE Signature 是 4 字节的标识 50 45 00 00（"PE\0\0"），用于验证这是一个合法的 PE 文件。

COFF Header 紧跟在 PE Signature 后面：

typedef struct _IMAGE_FILE_HEADER {
    WORD  Machine;              // 架构：IMAGE_FILE_MACHINE_I386(0x14c),
                                // IMAGE_FILE_MACHINE_AMD64(0x8664)
    WORD  NumberOfSections;     // Section 数量
    DWORD TimeDateStamp;        // 编译时间戳
    DWORD PointerToSymbolTable; // 符号表偏移（调试用）
    DWORD NumberOfSymbols;      // 符号数量
    WORD  SizeOfOptionalHeader; // Optional Header 大小
    WORD  Characteristics;      // 文件属性标志
} IMAGE_FILE_HEADER;

Characteristics 标志：

标志值含义
IMAGE_FILE_EXECUTABLE_IMAGE0x0002可执行文件
IMAGE_FILE_DLL0x2000DLL 文件
IMAGE_FILE_SYSTEM0x1000系统文件（驱动）
IMAGE_FILE_LARGE_ADDRESS_AWARE0x0020支持大于 2GB 地址空间
IMAGE_FILE_32BIT_MACHINE0x010032 位机器
IMAGE_FILE_DEBUG_STRIPPED0x0200调试信息已移除

5. Optional Header

虽然叫 "Optional"，但对于可执行文件来说是必须的。它包含加载器所需的关键信息。

// PE32+（64位）
typedef struct _IMAGE_OPTIONAL_HEADER64 {
    // Standard Fields
    WORD  Magic;                // 0x20b = PE32+, 0x10b = PE32
    BYTE  MajorLinkerVersion;
    BYTE  MinorLinkerVersion;
    DWORD SizeOfCode;
    DWORD SizeOfInitializedData;
    DWORD SizeOfUninitializedData;
    DWORD AddressOfEntryPoint;  // 入口点 RVA
    DWORD BaseOfCode;

    // Windows-Specific Fields
    ULONGLONG ImageBase;        // 首选基址
    DWORD SectionAlignment;     // 内存对齐（通常 0x1000）
    DWORD FileAlignment;        // 文件对齐（通常 0x200）
    WORD  MajorOperatingSystemVersion;
    WORD  MinorOperatingSystemVersion;
    WORD  MajorImageVersion;
    WORD  MinorImageVersion;
    WORD  MajorSubsystemVersion;
    WORD  MinorSubsystemVersion;
    DWORD Win32VersionValue;
    DWORD SizeOfImage;          // 内存中 PE 镜像总大小
    DWORD SizeOfHeaders;        // 所有 Header 的总大小
    DWORD CheckSum;
    WORD  Subsystem;            // 子系统：1=控制台, 2=GUI
    WORD  DllCharacteristics;
    ULONGLONG SizeOfStackReserve;
    ULONGLONG SizeOfStackCommit;
    ULONGLONG SizeOfHeapReserve;
    ULONGLONG SizeOfHeapCommit;
    DWORD LoaderFlags;
    DWORD NumberOfRvaAndSizes;  // 数据目录数量（通常 16）
    IMAGE_DATA_DIRECTORY DataDirectory[16]; // 数据目录
} IMAGE_OPTIONAL_HEADER64;

关键字段：

  • AddressOfEntryPoint：入口点的 RVA（相对虚拟地址），程序从这里开始执行
  • ImageBase：PE 文件的首选加载基址。64 位 EXE 通常是 0x140000000，DLL 通常是 0x180000000
  • SectionAlignment：内存中节的对齐粒度
  • FileAlignment：文件中节的对齐粒度
  • Subsystem：IMAGE_SUBSYSTEM_WINDOWS_CUI(3) = 控制台程序，IMAGE_SUBSYSTEM_WINDOWS_GUI(2) = 窗口程序

Data Directories（16 个）：

索引名称说明
0Export Directory导出函数表
1Import Directory导入函数表
2Resource Directory资源表（图标、对话框等）
3Exception Directory异常处理表（x64 结构化异常）
4Security Directory数字签名
5Base Relocation Table基址重定位表
6Debug Directory调试信息
7Architecture保留
8Global Ptr全局指针（RISC）
9TLS Directory线程局部存储
10Load Config Directory加载配置（SEH、CFG、GFID）
11Bound Import绑定导入
12IAT导入地址表
13Delay Import Descriptor延迟导入
14CLR Runtime Header.NET 元数据
15Reserved保留

6. Section Table（节表）

Section Table 紧跟在 Optional Header 后面，每个条目 40 字节，描述一个节的属性。

typedef struct _IMAGE_SECTION_HEADER {
    BYTE  Name[8];              // 节名（如 ".text"）
    union {
        DWORD PhysicalAddress;
        DWORD VirtualSize;      // 内存中实际大小
    } Misc;
    DWORD VirtualAddress;       // 内存中的 RVA
    DWORD SizeOfRawData;        // 文件中对齐后的大小
    DWORD PointerToRawData;     // 文件中的偏移
    DWORD PointerToRelocations;
    DWORD PointerToLinenumbers;
    WORD  NumberOfRelocations;
    WORD  NumberOfLinenumbers;
    DWORD Characteristics;      // 节属性标志
} IMAGE_SECTION_HEADER;

常见节及其作用：

节名内容属性
.text机器代码可执行、可读
.rdata只读数据（常量、导入表、导出表）可读
.data已初始化的全局变量可读、可写
.bss未初始化的全局变量可读、可写
.rsrc资源（图标、菜单、版本信息）可读
.reloc基址重定位信息可读
.edata导出表可读
.idata导入表可读
.tls线程局部存储可读、可写
.debug调试信息可读
.pdata异常处理信息（x64）可读
.gfidsControl Flow Guard 表可读

Characteristics 标志：

标志值含义
IMAGE_SCN_CNT_CODE0x00000020包含代码
IMAGE_SCN_CNT_INITIALIZED_DATA0x00000040包含已初始化数据
IMAGE_SCN_CNT_UNINITIALIZED_DATA0x00000080包含未初始化数据
IMAGE_SCN_MEM_EXECUTE0x20000000可执行
IMAGE_SCN_MEM_READ0x40000000可读
IMAGE_SCN_MEM_WRITE0x80000000可写
TIP
PE 中没有像 ELF 那样的 "段（Segment）" 概念。PE 的 Section 直接对应 ELF 的 Section。PE 的加载由 Windows PE Loader 直接按 Section Table 映射，不需要 Program Header Table 那样的中间层。

7. 核心数据目录

导入表与 IAT

导入表是 PE 最重要的数据结构之一，定义了程序依赖哪些外部 DLL 和函数。

// 导入描述符（每个被导入的 DLL 一个）
typedef struct _IMAGE_IMPORT_DESCRIPTOR {
    union {
        DWORD Characteristics;
        DWORD OriginalFirstThunk; // INT（Import Name Table）的 RVA
    };
    DWORD TimeDateStamp;
    DWORD ForwarderChain;
    DWORD Name;                   // DLL 名称的 RVA
    DWORD FirstThunk;             // IAT 的 RVA
} IMAGE_IMPORT_DESCRIPTOR;

INT vs IAT：

  • INT（Import Name Table）：存储函数名/序号的查找表，加载后不修改
  • IAT（Import Address Table）：存储实际函数地址的表，加载器填充真实地址

加载前 INT 和 IAT 的内容相同，加载后 IAT 被填充为实际地址。

加载前：
INT[0] → IMAGE_IMPORT_BY_NAME("MessageBoxA")
IAT[0] → IMAGE_IMPORT_BY_NAME("MessageBoxA")  // 相同

加载后：
INT[0] → IMAGE_IMPORT_BY_NAME("MessageBoxA")   // 不变
IAT[0] → 0x00007FFA12345678                    // MessageBoxA 的真实地址

每个函数的查找方式：

// IMAGE_IMPORT_BY_NAME
typedef struct _IMAGE_IMPORT_BY_NAME {
    WORD  Hint;     // 导出表中的猜测索引（加速查找）
    BYTE  Name[];   // 函数名字符串
} IMAGE_IMPORT_BY_NAME;

// 或者通过序号导入（高位为1时）
// 值 = 0x80000000 | Ordinal

导出表

导出表定义了 DLL 向外暴露的函数，供其他程序调用。

typedef struct _IMAGE_EXPORT_DIRECTORY {
    DWORD Characteristics;
    DWORD TimeDateStamp;
    WORD  MajorVersion;
    WORD  MinorVersion;
    DWORD Name;                 // DLL 名称 RVA
    DWORD Base;                 // 导出序号的基数（通常为 1）
    DWORD NumberOfFunctions;    // 导出函数总数
    DWORD NumberOfNames;        // 按名称导出的数量
    DWORD AddressOfFunctions;   // 函数地址表 RVA
    DWORD AddressOfNames;       // 函数名称表 RVA
    DWORD AddressOfNameOrdinals;// 名称序号映射表 RVA
} IMAGE_EXPORT_DIRECTORY;

查找导出函数的流程：

  1. 按名称：在 AddressOfNames 中二分查找函数名 → 得到索引 → 在 AddressOfNameOrdinals 中查序号 → 在 AddressOfFunctions 中取地址
  2. 按序号：序号 - Base = AddressOfFunctions 的索引 → 直接取地址

重定位表

当 PE 无法加载到首选基址（ImageBase）时，需要修正所有绝对地址引用。重定位表就是做这个的。

// 重定位块（按 4KB 页组织）
typedef struct _IMAGE_BASE_RELOCATION {
    DWORD VirtualAddress;   // 需要重定位的页 RVA
    DWORD SizeOfBlock;      // 此块的总大小
    // WORD TypeOffset[];   // 紧跟其后的重定位项数组
} IMAGE_BASE_RELOCATION;

// 每个重定位项：高 4 位 = 类型，低 12 位 = 页内偏移
// 类型：
// IMAGE_REL_BASED_ABSOLUTE (0)  — 填充项，无实际作用
// IMAGE_REL_BASED_HIGHLOW (3)   — 32 位重定位（PE32）
// IMAGE_REL_BASED_DIR64 (10)    — 64 位重定位（PE32+）

重定位的计算：真实基址 - 首选基址 = Delta。将 Delta 加到每个需要重定位的地址上。

WARN
EXE 文件通常有固定的 ImageBase（如 0x140000000），ASLR 开启时基址会随机化。DLL 的 ImageBase 只是建议值，ASLR 下会被随机分配到其他地址，所以 DLL 必须包含重定位表。

11. 加载与执行流程

1. 用户双击 EXE 或调用 CreateProcess
        │
2. Windows PE Loader (NtCreateSection + NtMapViewOfSection)
   ├── 验证 MZ 签名 → PE 签名
   ├── 检查 Machine 类型（x86/x64）
   ├── 检查 Subsystem（GUI/CUI）
   ├── 映射文件到内存（按 Section Table）
   │   ├── .text  → 可读可执行
   │   ├── .rdata → 只读
   │   ├── .data  → 可读可写
   │   └── .rsrc  → 只读
   ├── 加载依赖的 DLL
   │   ├── 读取 Import Directory
   │   ├── 递归加载每个 DLL
   │   └── 填充 IAT（Import Address Table）
   ├── 应用重定位（如果基址不对）
   ├── 设置 TLS 回调
   ├── 设置安全 cookie（/GS 保护）
   ├── 执行 DLL 的 DllMain（DLL_PROCESS_ATTACH）
   └── 跳转到 AddressOfEntryPoint
        │
3. CRT 入口（mainCRTStartup / wmainCRTStartup）
   ├── 初始化堆
   ├── 初始化 C 运行时
   ├── 获取命令行参数
   └── 调用 main() / wmain()
        │
4. 程序正常运行

DLL 加载顺序：

  1. 应用程序所在目录
  2. 系统目录（System32）
  3. 16 位系统目录（System）
  4. Windows 目录
  5. 当前目录
  6. PATH 环境变量中的目录

12. 地址转换：VA / RVA / File Offset

这是 PE 分析中最基础也是最常用的转换，请务必掌握。

当 FileAlignment 与 SectionAlignment 不同时，磁盘映像和内存映像不同，因此必须在每个区块内部进行转换。

RVA = VA - ImageBase
VA  = ImageBase + RVA

File Offset = RVA - (Section.VirtualAddress - Section.PointerToRawData)
           = RVA - Delta

其中 Delta = VirtualAddress - PointerToRawData（每个区块不同）

实例计算：

区块VirtualAddress (RVA)PointerToRawDataDelta
.text0x10000x4000xC00
.rdata0x20000x6000x1A00
.data0x30000x8000x2800
例 1：VA = 0x00401112（属于 .text 块）
File Offset = 0x00401112 - 0x00400000 - 0xC00 = 0x512

例 2：VA = 0x00402002（属于 .rdata 块）
File Offset = 0x00402002 - 0x00400000 - 0x1A00 = 0x602
实用工具
使用 LordPE 等工具，单击 "RLC" 按钮打开地址转换器，输入 VA 即可自动计算。建议实际分析时用工具辅助。

13. 资源的三层目录结构

资源用类似文件系统目录的方式保存，通常包含 3 层：

第1层（资源类型）    第2层（资源 ID）     第3层（代码页）    资源数据

[ICON (3)] ───────── [ID=101] ──────────── [英文] ──────────> 图标数据
                        ├───────────────── [中文] ──────────> 图标数据2
                        └───────────────── [日文] ──────────> 图标数据3

[MENU (4)] ──────────── [ID=200] ──────────── [英文] ──────────> 菜单数据
ID资源类型
1Cursor（光标）
2Bitmap（位图）
3Icon（图标）
4Menu（菜单）
5Dialog（对话框）
6String（字符串表）
14Version Info（版本信息）

核心结构：IMAGE_RESOURCE_DIRECTORY（目录节点）→ IMAGE_RESOURCE_DIRECTORY_ENTRY（目录条目）→ IMAGE_RESOURCE_DATA_ENTRY（数据入口）。资源名称字符串使用 Unicode 编码。资源是 PE 文件所有结构中最复杂的部分。

14. 调试目录（Debug Directory）

数据目录表 Index=6 指向调试目录，由 IMAGE_DEBUG_DIRECTORY 结构数组组成：

typedef struct _IMAGE_DEBUG_DIRECTORY {
    DWORD Characteristics;
    DWORD TimeDateStamp;
    WORD  MajorVersion;
    WORD  MinorVersion;
    DWORD Type;                 // 调试信息类型
    DWORD SizeOfData;           // 调试数据大小
    DWORD AddressOfRawData;     // 加载后的 RVA（为 0 表示不被映射）
    DWORD PointerToRawData;     // 文件偏移
} IMAGE_DEBUG_DIRECTORY;

最常见的调试信息存储方式是 PDB 文件：

  • Visual Studio 6.0：debug 头部以 "NB10" 标识开始
  • Visual Studio .NET & 后续版本：debug 头部以 "RSDS" 标识开始

15. 其他 PE 结构

TLS 回调（安全重点）

TLS（Thread Local Storage）回调函数在程序的 AddressOfEntryPoint 之前执行。数据目录 Index=9 指向 IMAGE_TLS_DIRECTORY，其中 AddressOfCallBacks 是一个函数指针数组。程序退出时 TLS 回调还会再执行一次。

typedef struct _IMAGE_TLS_DIRECTORY32 {
    DWORD StartAddressOfRawData;     // TLS 数据起始（VA，不是 RVA!）
    DWORD EndAddressOfRawData;
    DWORD AddressOfIndex;            // TLS 索引
    DWORD AddressOfCallBacks;        // 回调函数指针数组 ★
    DWORD SizeOfZeroFill;
    DWORD Characteristics;
} IMAGE_TLS_DIRECTORY32;
注意
• IMAGE_TLS_DIRECTORY 中的地址是 VA 而不是 RVA！
• 该结构本身不在 .tls 区块中，而在 .rdata 区块中。
• TLS 回调可用于在入口点之前执行反调试代码，是软件保护的重要手段。

绑定输入（Bound Import）

绑定输入是对输入表的一种优化：在安装程序时预先把 DLL 的真实地址计算好写入 IAT，以加快程序启动速度。Index=11 指向绑定输入表。如果绑定成功但 DLL 版本不匹配，加载器会自动回退到正常导入流程。

延迟载入（Delay Load Import）

延迟载入通过链接器和运行库加入额外代码实现——函数首次被调用时才加载 DLL。Index=13 指向延迟载入描述符。
