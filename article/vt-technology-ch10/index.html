1. 硬件虚拟化的基本概念

初学者提示
VT（Virtualization Technology）是 Intel 的硬件辅助虚拟化技术。它引入了一个全新的 CPU 层级——Ring -1，这个层级的权限比操作系统所在的 Ring 0 还要高。如果你之前只听说过 Ring 0（内核态）和 Ring 3（用户态），那么 VT 技术就是在 Ring 0 之下再挖一层，这一层的代码叫 Hypervisor（虚拟机监控器），拥有"上帝视角"。

VT 技术最初是为了提高 VMware 之类软件虚拟化的性能而设计的。但随着安全研究人员的探索，人们很快发现——由于 VT 引入了新的 CPU 层级 Ring -1，它在安全领域也带来了大量的新应用场景。

1.1 概述

硬件虚拟化技术引入了新的 CPU 模式和虚拟化指令集，帮助 VMM（Virtual Machine Monitor，虚拟机监控器）提升性能。Intel VT-x 为 CPU 提供了 VMX（Virtual Machine Extension，虚拟机扩展）功能。

在 VMX 中有两种操作模式：

  • VMX Root Operation（Root 模式）—— 供 VMM/Hypervisor 使用，拥有最高权限
  • VMX Non-Root Operation（Non-Root 模式）—— Guest OS（客户操作系统）运行的模式

VMM 通过虚拟机控制结构（Virtual-Machine Control Structure，VMCS）设置需要截获的硬件访问，例如指令、中断、内存访问和异常等。如果 Guest OS 在执行时触发了这些条件设置，将引起 #VMExit（VM 退出），此时 CPU 会从 Non-Root 模式切换到 Root 模式，VMM 获得控制权并执行相应的处理程序。当 VMM 执行结束后，将控制权返回给 Guest OS，这称为 #VMEntry（VM 进入）。

类比理解
编写一个 VT 技术的 Hypervisor 例子和编写 Windows 窗口程序非常相似：
1. 分配 VMXON 区域和 VMCS 控制块（类似 RegisterClassEx 注册窗口）
2. 填写 VMCS 控制块，设置要监控什么特权指令、是否开启 EPT 等（类似填写窗口过程 WndProc）
3. 调用 VMLAUNCH 指令启动虚拟机（类似 CreateWindow 显示窗口）
4. 当产生 #VMExit 事件时调用 ExitEventHandler 函数处理（类似在 WndProc 中处理消息）

VT 技术并不神秘，与编写窗口程序有很多相似之处。其难点在于调试（出现错误时定位比较困难）。

1.2 相关结构和汇编指令

为了实现 VT 技术，Intel 引入了一系列新的指令集：

指令功能说明
VMXON开启 VMX 模式执行后续虚拟化相关指令的前提
VMXOFF关闭 VMX 模式后续虚拟化指令执行都会失败
VMLAUNCH启动虚拟机启动 VMCS 指向的 Guest OS，触发 #VMEntry
VMRESUME恢复虚拟机从 Hypervisor 中恢复 Guest OS 的执行
VMPTRLD激活 VMCS加载一个 VMCS 指针，后续操作针对该 VMCS
VMCLEAR清除 VMCS使一块 VMCS 变为非激活状态
VMREAD读 VMCS读取当前 VMCS 中的数据
VMWRITE写 VMCS向当前 VMCS 中写入数据
VMCALLGuest-Hypervisor 交互Guest OS 中执行，产生 #VMExit 陷入 Hypervisor
INVEPT刷新 EPT 缓存使 TLB 中缓存的 EPT 地址映射失效

VT 的基本工作流程：

  1. 使用 VMXON 进入 VMX Root 模式（Hypervisor 模式）
  2. 对 Guest OS 进行配置（填写 VMCS）
  3. 使用 VMLAUNCH 从 Root 模式转到 Non-Root 模式（Guest OS）
  4. Guest OS 运行中产生 #VMExit 事件，陷入 Hypervisor（Root 模式）
  5. Hypervisor 处理事件
  6. 调用 VMRESUME 重新 #VMEntry 到 Guest OS
  7. 如此循环，直到调用 VMXOFF 关闭 VMX 模式

VMCS（Virtual Machine Control Structure）

关键理解
VMCS 是 VT 技术中最重要的数据结构。可以简单地认为——每个 VMCS 代表一个虚拟机。通过控制 VMCS 就能控制虚拟机的各种行为和属性。CPU 支持多个 Guest OS，每个 Guest OS 对应一个 VMCS。

VMCS 的数据区分为以下 6 个部分：

#区域名称作用
1Guest-State Area
客户区状态域		#VMEntry 时加载 Guest OS 状态；#VMExit 时保存 Guest OS 状态。包括通用寄存器、CR0/CR3/CR4、RSP/RIP、段寄存器、GDTR/IDTR 等。
2Host-State Area
宿主机状态域		#VMExit 时加载 Hypervisor 的状态，恢复 Hypervisor 的执行环境。
3VM-Execution Control Fields
虚拟机执行控制域		最关键的配置区域。设置各种退出条件：控制哪些事件会导致 #VMExit，包括异常、I/O 访问、MSR 访问、EPT 配置等。
4VM-Entry Control Fields控制 #VMEntry 时的行为，如是否进入 x64 模式、是否注入中断等。
5VM-Exit Control Fields定义 #VMExit 后硬件立即执行的操作。
6VM-Exit Information Fields只读区域。包含最近一次 #VMExit 的退出原因、指令地址等信息。处理 #VMExit 时必须使用此处信息才能知道陷入原因。
学习建议
第一次接触 VT 时，不要纠缠于每一个字段的细节。先在大局上理解 VT 技术的运作方式——Root/Non-Root 模式的切换、VMCS 的 6 个区域各自的作用，然后再深入研究代码细节。Intel 开发者手册（Intel SDM）是最终的权威参考。

1.3 EPT 机制（扩展页表）

初学者提示
EPT 是 VT 技术中与安全关系最密切的特性之一。简单理解，EPT 就是多了一层地址转换——在原有的 CR3 页表转换之外，再加一层 EPT 页表。这层额外的转换给了 Hypervisor 巨大的控制空间，可以实现内存隐藏、内存访问监控等强大的安全功能。

在虚拟化平台上，每个 Guest OS 都有独立的物理地址空间，地址转换流程如下：

  1. Guest OS 中的虚拟地址（GVA）→ 通过 CR3 页表 → GPA（客户机物理地址）
  2. GPA → 通过 EPT（由 EPTP 寄存器配置）→ HPA（宿主机物理地址）

EPT 页表最多包含 4 级：

级别表项映射粒度
1（PML4T）EPT PML4E512 GB
2（PDPT）EPT PDPTE1 GB（大页面）或指向下级
3（PDT）EPT PDE2 MB（大页面）或指向下级
4（PT）EPT PTE4 KB（普通页面）

EPT 页表项的关键属性位：

位名称含义
0Read可读
1Write可写
2Execute可执行
5:3Memory Type内存类型（如 WB=6, UC=0）
7Page Size大页面标志（仅 PDPTE/PDE 有效）
EPT Violation
如果从 GPA 转换到 HPA 的过程中发生错误，将引发两类 EPT 故障——EPT Violation 和 EPT Misconfiguration。当发生这些故障时，处理器会产生 #VMExit 事件（退出原因为 EXIT_REASON_EPT_VIOLATION）。Hypervisor 在 #VMExit 处理中分析具体原因，修正页表映射。这是实现内存隐藏等技术的基础。

2. VT 技术的应用

本节目标
通过分析一个完整的 Shadow Walker 内存隐藏实例，理解 VT 技术在实际安全场景中的使用方法。这个例子展示了如何利用 EPT 机制，让同一个内存地址在代码执行和数据访问时得到截然不同的结果。

2.1 ShadowWalker 实例

Shadow Walker 通过 EPT 机制将 Split.exe 程序的代码执行和数据访问映射到不同的物理地址上。这样，Split.exe 在执行指令和读取数据时就会得到不同的结果，从而达到隐藏内容的目的。

Split.exe 的代码很简单：首先将 PrintValue 的值作为代码执行一遍，然后打印 PrintValue 的值。原始值 0x90902390 作为代码执行是没问题的，但打印的结果却是 0x79696450。秘密就在 Hypervisor.sys 里面——Hypervisor 将 NewSeon 这个区段根据代码执行和数据访问进行了不同的映射。

示例程序的限制条件：

  • 机器必须支持 Intel VT-x 技术及 EPT 特性，并在 BIOS 中开启 VT
  • 操作系统必须是 32 位 Windows 7
  • CPU 关闭了多核支持（单核模式）
  • 关闭 PAE 机制（bcdedit /set pae forcedisable）
注意
Hypervisor.sys 属于内核模块，一旦造成蓝屏可能会损坏系统，建议在 VMware 虚拟机中运行。VMware Workstation 10 以上的版本都支持 VT 技术的模拟。

分析 Hypervisor

Hypervisor 内核模块的大体功能：

  1. 在 DriverEntry 中检测 CPU 对 VT 的支持情况
  2. 在 CPU 支持 VT 技术的情况下，配置 VMCS 和 EPT 信息，开启 VT
  3. 通过 cpuid 测试 VT 的效果（修改 cpuid 的返回值以验证 VT 已生效）
  4. 注册进程创建回调，监控 Split.exe 的创建
  5. 当 Split.exe 创建时，将其内存复制为两份：一份作为数据访问，另一份作为代码执行
  6. 对数据访问拷贝进行修改（将 PrintValue 值修改为 "Pdiy"）
  7. 两份拷贝都进行 EPT 映射，根据是代码执行还是数据访问，映射到不同的 HPA
  8. 在 Split.exe 退出时拆除页面分割并释放内存
  9. 驱动被卸载时，删除进程回调并关闭 VT

例如，修改 cpuid 功能号 0xPdiy 的处理：

// 开启 VT 前：cpuid 0xPdiy 返回全 0
// 开启 VT 后：Hypervisor 截获并修改返回值
if (Function == 0xPdiy) {
    GuestReg->RegEax = 0x11111111;
    GuestReg->RegEcx = 0x22222222;
    GuestReg->RegEdx = 0x33333333;
    GuestReg->RegEbx = 0x49144444;
}
原理总结
只要系统运行在 Non-Root 模式下，任何 cpuid 指令的执行都会触发 #VMExit 陷入 Hypervisor。Hypervisor 的 #VMExit 处理函数可以检查退出原因，如果是 cpuid 指令并且功能号是感兴趣的，就可以修改返回值再回到 Guest OS。这就是 VT 反调试、VT 保护技术的核心思想。

2.2 检测 VT 支持情况

在开启 VT 之前，必须检查 CPU 和相关硬件是否满足条件：

  1. 检测 CPU 是否支持 VMX：执行 CPUID.1:ECX[5]，检查 ECX 第 5 位
  2. 检测 BIOS 是否开启了 VT：读取 IA32_FEATURE_CONTROL MSR（0x3A）
  3. 检测 Secondary ProcBased Control：读取 IA32_VMX_PROCBASED_CTLS（0x482）
  4. 检测 EPT 支持：读取 IA32_VMX_PROCBASED_CTLS2（0x48B）
// 检测 VMX 是否支持（CPUID.1:ECX[5]）
ExecuteCpuid(0x1, &Eax, &Ebx, &Ecx, &Edx);
if (!(Ecx & (1 << 5))) {
    KdPrint("VMX not supported\n");
    return STATUS_UNSUCCESSFUL;
}

// 检测 BIOS 是否开启了 VT
Msr = ReadMsr(IA32_FEATURE_CONTROL);
if (!(Msr & (1 << 2))) {
    KdPrint("VMX not enabled in BIOS\n");
    return STATUS_UNSUCCESSFUL;
}

// 检测 EPT 支持
ReadMsr(IA32_VMX_PROCBASED_CTLS2, &Ctls2);
if (!(Ctls2.EnableEpt && Ctls2.EnableVpid)) {
    KdPrint("EPT/VPID not supported\n");
    return STATUS_UNSUCCESSFUL;
}

当所有条件满足后，将 CR4.VMXE 置位。只有设置了 CR4 的 VMXE 位，才能执行后续的 VT 指令。

2.3 VMCS 的配置

将 CR4.VMXE 置位后，就可以进入开启 VT 的流程了。整个过程在 LoadHypervisor() 函数中完成：

  1. 为每个 CPU 核心分配 CPU_VMX_CONTEXT 结构，存放 VMCS、IOBitmap、ExceptionBitmap 和 EPT 信息
  2. 调用 SetupVMX() 执行 VMXON 指令，进入 VMX Root 模式
  3. 调用 SetupVMCS() 填充 VMCS 结构的 6 个部分

Guest-State Area（客户区状态域）

如果不是为了编写类似 VMware 的虚拟机，只是把当前操作系统放入 Guest OS 中执行，那么 Guest OS 的配置基本上要和 Hypervisor 的一样。需要填写的内容包括：CR0/CR3/CR4、DR7、RSP、RIP、RFLAGS、段寄存器等。

其中最关键的是 GUEST_EIP——从 Hypervisor（Root 模式）切换到 Guest OS（Non-Root 模式）时，CPU 会从 GUEST_EIP 填写的地址处开始执行。

Host-State Area（宿主机状态域）

关键区别在于 HOST_EIP——这是 #VMExit 事件发生时 CPU 跳转到的地址，相当于 VT 技术中的"窗口过程"。

// 设置 Host ESP 和 EIP
WriteVmcs(HOST_ESP, (ULONG)CpuInfo->HostEsp - 0x7000);

// HOST_EIP 是关键！产生 #VMExit 时进入这里
WriteVmcs(HOST_EIP, (ULONG)ExitEventHandler);

VM-Execution Control Fields（虚拟机执行控制域）

这是 VT 配置中最重要的区域：

// 配置异常位图 - 设置哪些异常需要截获
ExceptionBitmap = 0;
ExceptionBitmap |= (1 << DEBUG_EXCEPTION);    // 调试异常
ExceptionBitmap |= (1 << BREAKPOINT_EXCEPTION); // 断点异常
ExceptionBitmap |= (1 << PAGE_FAULT_EXCEPTION); // 页异常
WriteVmcs(EXCEPTION_BITMAP, ExceptionBitmap);

// 配置 I/O 位图 - 拦截 I/O 端口访问
WriteVmcs(IO_BITMAP_A, CpuInfo->IOBitmapPhysicalAddr.LowPart);
WriteVmcs(IO_BITMAP_B, CpuInfo->IOBitmapPhysicalAddr.HighPart);

// 配置 MSR 位图 - 拦截 MSR 寄存器访问
WriteVmcs(MSR_BITMAP, CpuInfo->MSRBitmapPhysicalAddr.LowPart);

2.4 EPT 的配置

EPT 配置两步走
1. 设置 EPT 指针（EPTP），指向 EPT 页表
2. 初始化 EPT 页表结构

首先在 VMCS 的 CPU_BASED_VM_EXEC_CONTROL 字段中启用 Secondary Controls，然后在 SECONDARY_VM_EXEC_CONTROL 字段中将 ENABLE_EPT 位置 1：

// 配置 EPT 指针
PhysicalAddress = MmGetPhysicalAddress(CpuInfo->EptInfo.EptPml4TablePointer);
EptPointer.Bits.PhysAddr = PhysicalAddress.LowPart >> 12;
EptPointer.Bits.PageWalkLength = 0;
WriteVmcs(EPT_POINTER, EptPointer.UnsignedVal);

// 启用 EPT - bit 1 置 1
WriteVmcs(SECONDARY_VM_EXEC_CONTROL, (1 << 1));

// 激活 Secondary Controls
WriteVmcs(CPU_BASED_VM_EXEC_CONTROL, Value | (1 << 31));

EPT 页表的初始化（由 InitEptIdentityMap() 实现）：

// 32 位系统，4GB 内存空间
// 分配 1 个 PML4T（512 项）
Pml4Ptr = AllocateContiguousMemory(512 * sizeof(EPT_PML4_ENTRY));

// 分配 4 个 PDPT（每项管理 1GB）
for (i = 0; i < 4; i++) {
    PdptePtr[i] = AllocateContiguousMemory(512 * sizeof(EPT_PDPTE_ENTRY));
}

// 设置 PML4E：可读、可写、可执行
Pml4Ptr->Present = 1;
Pml4Ptr->Write = 1;
Pml4Ptr->Execute = 1;

// 设置 PDE 使用 2MB 大页面（bit 7 = 1）
PdePtr->Size = 1;
PdePtr->Present = 1;
PdePtr->Write = 1;
PdePtr->Execute = 1;
PdePtr->MemoryType = EPT_MEMORY_TYPE_WB;
EPT 页表内存布局
在 32 位系统中最多支持 4GB 内存：
- 1 个 PML4T 管理 512GB 空间
- 4 个 PDPTE 管理 4GB 空间（每个 1GB）
- 512 个 PDE，每个使用 2MB 大页面
- 没有使用 PTE（因为 PDE 的 Page Size 位已置 1）

即 2MB × 512 × 4 = 4GB，覆盖了整个 32 位地址空间。

2.5 开启 VT

准备工作完成后，执行 VMLAUNCH 指令开启 VT：

NTSTATUS LaunchVirtualize(PCPU_VMX_CONTEXT CpuInfo)
{
    // VMLAUNCH 不应该返回，如果成功，将进入 VMCS 中的 Guest EIP
    ExecuteVmLaunch();

    if (VmFailInvalid()) {
        KdPrint("VMLAUNCH Failed\n");
        return STATUS_UNSUCCESSFUL;
    }

    if (VmLaunchFailValid()) {
        KdPrint("VMLAUNCH Failed, Error Code: %p\n",
                ReadVmcs(VM_INSTRUCTION_ERROR));
        return STATUS_UNSUCCESSFUL;
    }

    return STATUS_UNSUCCESSFUL;
}

如果 VT 开启成功，VMLAUNCH 不会返回，CPU 会直接跳转到 VMCS 中设置的 GUEST_EIP 地址开始执行。

2.6 内存隐藏的实现

核心思想
EPT 机制可以对同一个 GPA 根据访问类型（代码执行还是数据读写）映射到不同的 HPA。这就是 VT 技术实现内存隐藏的根本原理。

内存隐藏的具体实现流程：

  1. 监控进程创建：调用 PsSetCreateProcessNotifyRoutine 注册进程创建回调
  2. 复制 PE 文件：当检测到 Split.exe 创建时，将其整个 PE 文件复制一份——原始页面用于代码执行，复制的页面用于数据访问
  3. 修改数据副本：对数据访问副本中 NewSeon 节的 PrintValue 值进行修改
  4. 通过 VMCALL 通知 Hypervisor：发送 VMCALL_INIT_SPLIT 编号的 #VMExit 事件
  5. 使 EPT 页面无效：在 Hypervisor 中遍历 Split.exe 的所有 CODE_EPT 页面，将对应的 EPT PTE 的 Present 位置 0
  6. 捕获 EPT Violation：当 Split.exe 访问这些页面时，产生 EXIT_REASON_EPT_VIOLATION
  7. 动态映射：在 #VMExit 处理函数中，根据访问类型进行不同的 GPA 到 HPA 映射
// EPT Violation 处理 - 根据访问类型映射到不同物理地址
ExitQualification = ReadVmcs(EXIT_QUALIFICATION);

if (ExitQualification & EPT_MSK_DATA_EXEC) {
    // 代码执行访问 → 映射到 CodePhys
    PtePtr->PhysAddr = TranslationPtr->CodePhys >> 12;
    PtePtr->Execute = 1;
}
else if ((ExitQualification & EPT_MSK_DATA_READ) ||
         (ExitQualification & EPT_MSK_DATA_WRITE)) {
    // 数据访问 → 映射到 DataPhys
    PtePtr->PhysAddr = TranslationPtr->DataPhys >> 12;
    PtePtr->Present = 1;
    PtePtr->Write = 1;
}
内存隐藏的应用场景
- 反调试：让调试器读取到的代码与 CPU 实际执行的代码不同
- 代码保护：加密的代码在数据访问时不可见，只有 CPU 执行时才能解密
- Rootkit 隐藏：隐藏恶意代码的踪迹，安全软件无法扫描到
- 游戏保护：防止内存修改器篡改游戏数据

理解了这个原理，就能明白为什么 VT 技术在软件保护领域如此重要。

3. VT 调试方法

调试策略
VT 技术本身并不复杂，只是因为参考资料较少、难以调试，才给人造成研究困难的假象。使用 VMware + WinDbg + IDA Pro 进行调试，可以让难度大幅降低。

双机调试环境搭建

  1. 选择 VMware 版本：VMware Workstation 8 以上版本支持 VT 技术程序，推荐 Workstation 12
  2. BIOS 开启 VT-x：在 BIOS 中开启 VT-x 技术支持
  3. 配置虚拟化引擎：在虚拟机"处理器"设置中，将"首选模式"改成 Intel VT-x/EPT
  4. 使用 VirtualKD：下载 VirtualKD，将 target 目录复制到被调试机中，运行 vminst.exe 即可自动完成配置

Root 模式调试（Hypervisor 代码）

使用 WinDbg 无法调试进入 VT 后 Root 模式下的代码。要调试 Hypervisor 代码，需要使用 IDA Pro + VMware 的 GDB Stub 功能：

// 在虚拟机 .vmx 文件中添加以下配置
debugStub.listen.guest32 = "TRUE"
debugStub.listen.guest64 = "TRUE"
monitor.debugOnStartGuest32 = "TRUE"
debugStub.hideBreakpoints = "TRUE"
bios.bootDelay = "3000"

GDB Stub 分为两部分：当虚拟 CPU 运行于 32 位模式时，在 8832 端口监听；当运行于 x64 模式时，在 8864 端口监听。

配置完成后：

  1. 在 IDA Pro 中选择 "Debugger" → "Attach" → "Remote GDB Debugger"
  2. 设置 Hostname 为 "localhost"，端口为 8832
  3. 选择 PID 为 0 的进程进行附加
  4. 在 IDA Pro 中加载 Hypervisor.sys 的 PDB 符号文件
双调试器协同工作
- WinDbg：调试 Non-Root 模式（Guest OS 内核代码）
- IDA Pro：调试 Root 模式（Hypervisor 代码）

Hypervisor（Root 模式）中的代码通常不会太多，IDA Pro 加上调试符号基本可满足调试需求。Non-Root 模式使用 WinDbg 效果更好。

4. 总结

概念核心要点安全意义
VT / VMXIntel 硬件虚拟化技术，引入 VMX Root 和 Non-Root 两种操作模式创建 Ring -1 特权层，权限高于操作系统 Ring 0
Hypervisor运行在 Root 模式的代码，相当于 VMM拥有"上帝视角"，可监控操作系统的各种行为
VMCS虚拟机控制结构，6 个区域，每个 VMCS 代表一个虚拟机通过 VMCS 控制 #VMExit 的触发条件
#VMExit / #VMEntryRoot 与 Non-Root 模式的切换事件所有特权操作都可能触发 #VMExit
EPT扩展页表，在 CR3 页表之上增加 GPA → HPA 转换可对同一 GPA 根据访问类型映射到不同 HPA
EPT ViolationEPT 地址转换失败时触发的 #VMExit 事件Hypervisor 可动态修正映射
VMCALLGuest OS 与 Hypervisor 的交互指令Guest OS 主动触发 #VMExit 进入 Hypervisor
Shadow Walker利用 EPT 实现代码/数据不同映射的内存隐藏示例反调试、代码保护、Rootkit 隐藏、游戏保护
VT 在逆向工程中的重要性
随着越来越多的软件保护方案（如 VMProtect、Themida、腾讯游戏保护等）采用 VT 技术，理解 VT 原理已经成为了高端逆向工程师的必备技能。VT 技术虽然入门门槛较高，但其核心思想——通过硬件虚拟化实现比操作系统更高的权限——并不复杂。关键在于多调试、多实践，逐步积累经验。
