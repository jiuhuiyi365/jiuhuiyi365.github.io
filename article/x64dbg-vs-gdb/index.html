1. 为什么要了解 x64dbg

长期使用 GDB 调试 Linux 程序的逆向工程师，转到 Windows 平台时会面临一个核心问题：GDB 在 Windows 上存在感极弱。虽然 MinGW 的 GDB 可以调试 PE 文件，但缺少对 Windows 内部机制（PEB、TEB、SEH、Wow64）的原生支持，调试体验大打折扣。

x64dbg 是目前 Windows 平台上最成熟的开源调试器，它的定位和 OllyDbg 类似，但支持 64 位程序，且插件和脚本生态更丰富。掌握 x64dbg 的全部能力，才能在 Windows 逆向中发挥 100% 的效率。

2. 核心差异对比

架构与定位

TIP
x64dbg 不是 WinDbg 的替代品。内核调试、驱动分析、崩溃转储分析仍然需要 WinDbg。x64dbg 专注的是用户态的 PE 文件调试。

操作方式差异

GDB 用户需要适应的最大变化：

1. 从命令驱动到 GUI + 命令混合

GDB 的一切操作都通过命令完成：break main、ni、x/10gx $rsp。x64dbg 虽然也有命令行窗口，但大部分操作可以通过鼠标完成：

  • 双击反汇编窗口的指令 → 设置断点
  • 右键寄存器值 → 在内存中跟随
  • 右键栈地址 → 在转储中查看
  • 鼠标悬停 → 显示符号解析结果

3. 寄存器差异

GDB 常用寄存器x64dbg 对应说明
$raxRAX通用寄存器，返回值
$rspRSP栈指针
$ripRIP指令指针
$rdi, $rsiRDI, RSI函数参数1、2（Linux）
-RCX, RDX, R8, R9函数参数1-4（Windows x64 调用约定）
-GS (x64) / FS (x86)指向 TEB，Windows 特有
WARN
Windows x64 使用 Microsoft x64 调用约定，前 4 个参数通过 RCX、RDX、R8、R9 传递，和 Linux System V ABI（RDI、RSI、RDX、RCX）完全不同。这是 GDB 用户最容易搞混的地方。

3. x64dbg 的窗口布局与工作流

x64dbg 启动后默认分为 5 个核心面板：

  • 反汇编窗口（CPU）：左上方，显示当前指令及其上下文，是主要工作区
  • 寄存器窗口：右上方，实时显示所有寄存器值，支持右键菜单修改
  • 栈窗口：右下方，显示当前栈帧内容
  • 内存转储窗口（Dump）：左下方，十六进制内存查看器
  • 日志窗口：底部，显示调试器日志和脚本输出

此外通过菜单可以打开：内存映射（Memory Map）、断点管理器、句柄、线程、模块、SSE 寄存器、调用栈等窗口。

GDB 用户建议的调试工作流：

  1. 拖入 PE 文件或附加（Attach）到进程
  2. 在符号/模块窗口中确认目标 DLL 已加载
  3. 用 bp 函数名 或 bp 地址 在命令行设置断点（和 GDB 风格一致）
  4. 运行到断点后，用 GUI 查看反汇编、寄存器、栈、内存
  5. 用 sti（步入）、sto（步过）、run（运行）命令控制执行

4. 断点系统详解

x64dbg 的断点系统比 GDB 更细粒度：

断点类型GDBx64dbg用途
软件断点break *addr（INT3）双击指令（INT3）常规断点
硬件断点hbreak *addr右键 → 断点 → 硬件执行不可被软件检测
内存断点watch *(int*)addr右键内存 → 内存断点监控整个内存页
条件断点break *addr if expr断点属性 → 条件表达式按条件触发
日志断点-断点属性 → 日志命令命中时只记录不暂停
DLL 断点set stop-on-solib-events 1选项 → 事件 → DLL 加载/卸载监控模块加载
API 断点break MessageBoxAbp MessageBoxA（自动下符号）拦截 Win32 API
TIP
x64dbg 的日志断点（Log breakpoint）是 GDB 没有的能力。设置后每次命中只在日志窗口输出指定信息而不暂停程序，非常适合批量记录函数调用参数。
加密调试提醒
软件断点通过将指令首字节改为 0xCC（INT3）实现，这会修改内存代码。调试加密/加壳程序时，修改内存可能导致解密后的代码校验不通过或数据错误。硬件断点（CPU 调试寄存器 DR0-DR3）不修改任何内存，仅由 CPU 在指令执行时检查，在加密相关调试中极其有用。

5. 内置分析与脱壳能力

这是 x64dbg 相比 GDB 最大的优势领域：

自动分析（Analysis）

x64dbg 可以自动分析二进制代码，识别函数边界、参数个数、局部变量、循环结构等。在反汇编窗口中右键 → "分析模块"，或者命令行输入 analr 即可。分析后的反汇编结果会标注函数名、参数、交叉引用等信息。

内置汇编器

在反汇编窗口选中一条指令，按 Space（或右键 → 汇编），可以直接输入汇编指令替换当前指令。GDB 中修改指令需要先用 set *(unsigned int*)addr = 0x90 这样的方式手动写十六进制，x64dbg 则直接支持汇编语法：

; 直接在 x64dbg 中输入
nop
mov eax, 1
ret

Scylla 脱壳工具

x64dbg 内置了 Scylla（通过插件形式），专门用于：

  • IAT 重建：脱壳后的程序导入表被破坏，Scylla 可以自动修复
  • Dump 内存：将进程内存完整 dump 到文件
  • 入口点修复：还原程序原始入口点

Trace 功能

x64dbg 内置两种 Trace 模式：

  • Trace record：记录哪些指令被执行过（类似 GDB 的 record full），在反汇编窗口中用不同颜色标记
  • Run trace：记录完整的执行路径（指令地址、寄存器状态），可以回放查看

6. 脚本系统

这是 GDB 用户最关心的部分。x64dbg 提供三层扩展能力：

命令脚本

x64dbg 支持类似 OllyDbg 的命令脚本（.txt 文件），语法简洁，适合批量操作：

// x64dbg 命令脚本示例：批量下断点并记录
var addr
mov addr, 0x00401000

loop:
cmp addr, 0x00402000
je done
bp addr
log “Set breakpoint at {addr}”
add addr, 4
jmp loop


done:
msg “All breakpoints set!”
ret

支持的指令包括：mov、add、sub、cmp、je/jne/jmp、bp、bc、log、msg、ret 等。

JavaScript 脚本

x64dbg 内置 JavaScript 引擎（Duktape），脚本文件后缀 .js，功能比命令脚本强大得多：

// x64dbg JS 脚本：监控 CreateFileA 调用并记录文件名

// 下断点
var addr = “CreateFileA”;
bp(addr);


// 注册断点回调
var hitCount = 0;


// 注册命令回调
addCommand(“logcall”, function() {
var rcx = regRead(“RCX”); // 第一个参数：文件名指针
var filename = str(rcx);  // 读取字符串
log(“CreateFileA called: ” + filename);
return true;
});


log(“Script loaded. Waiting for CreateFileA…”);

JS API 主要函数：

分类函数说明
寄存器regRead(name) / regWrite(name, val)读写寄存器
内存memRead(addr, size) / memWrite(addr, data)读写内存
断点bp(addr) / bc(addr)设置/清除断点
调试run() / sti() / sto()执行控制
模块modbase(name) / modsize(name)获取模块基址/大小
符号symfromaddr(addr)地址解析为符号名
输出log(msg) / msg(text)日志输出 / 弹窗
回调addCommand(name, cb) / addCallback(event, cb)注册命令/事件回调
TIP
如果你更习惯 Python，可以通过第三方项目 x64dbg-python（pip install x64dbg）在外部 Python 进程中通过调试接口控制 x64dbg。不过原生能力上，JavaScript 是官方支持的脚本语言。

C++ 插件开发

对于需要最高性能和最深度集成的场景，x64dbg 提供 C++ SDK 开发插件。插件以 DLL 形式加载，可以：

  • 注册自定义命令（和内置命令一样使用）
  • 拦截所有调试事件（断点、异常、模块加载、线程创建等）
  • 访问完整的调试器 API（内存、寄存器、符号、断点管理）
  • 添加菜单项、工具栏按钮、自定义窗口

插件基本框架：

// x64dbg 插件基础模板
#include "pluginsdk\_x64dbg.h"

// 插件入口
bool pluginInit(PLUG_INITSTRUCT* initStruct) {
// 注册自定义命令
_plugin_registercommand(
pluginHandle,
“MyPlugin”,
[](int argc, char* argv[]) {
dprintf(“MyPlugin command executed!\n”);
return true;
},
false
);
return true;
}


// 插件卸载
void pluginStop() {
_plugin_unregistercommand(pluginHandle, “MyPlugin”);
}

已有的优质 C++ 插件大多开源，可以学习其源码来掌握插件开发模式。

IMPORTANT
插件安装后需要在菜单 Plugins 中确认已加载，或重启 x64dbg。部分插件有 32 位和 64 位版本之分，必须和 x64dbg/x32dbg 的版本匹配。

8. 实战示例

自动 Hook 提取函数参数

假设目标程序调用了 MessageBoxA，我们想自动记录每次调用时的文本内容：

// 文件：hook_messagebox.js
// 功能：自动记录 MessageBoxA 的调用参数

// MessageBoxA 签名：
// int MessageBoxA(HWND hWnd, LPCSTR lpText, LPCSTR lpCaption, UINT uType)


// 下断点
bp(“MessageBoxA”);


// 注册断点命中的回调
addCallback(“breakpoint”, function() {
var rip = regRead(“RIP”);
var name = symfromaddr(rip);


if (name === "MessageBoxA") {
    var rcx = regRead("RCX");  // hWnd
    var rdx = regRead("RDX");  // lpText
    var r8  = regRead("R8");   // lpCaption
    var r9  = regRead("R9");   // uType
    
    var text = str(rdx);
    var caption = str(r8);
    
    log("=== MessageBoxA Called ===");
    log("Caption: " + caption);
    log("Text: " + text);
    log("Type: " + r9.toString(16));
}


});

批量断点与日志

在逆向分析中，经常需要在多个地址设置断点并记录执行路径。用命令脚本可以快速完成：

// 文件：batch_breakpoints.txt
// 功能：批量设置断点，命中时记录并自动继续

// 清除所有现有断点
bpc


// 设置断点
bp 0x00401000
bp 0x00401050
bp 0x004010A0
bp 0x00401100


// 设置日志断点（不暂停，只记录）
bpl 0x00401000, “Reached main entry”
bpl 0x00401050, “Reached init function”
bpl 0x004010A0, “Reached network call”
bpl 0x00401100, “Reached cleanup”


log “All breakpoints configured. Run the program.”

对比 GDB 中的等价操作：

# GDB 版本
break *0x00401000
commands
  silent
  printf "Reached main entry\n"
  continue
end

break *0x00401050
commands
silent
printf “Reached init function\n”
continue
end

x64dbg 的 bpl（log breakpoint）语法更简洁，不需要逐个写 commands...end 块。

Trace 记录与回溯

分析一段未知的代码执行路径时，x64dbg 的 Trace 功能非常有用：

// 1. 在感兴趣的位置下断点
bp 0x00401500
run

// 2. 到达断点后，开启 Run Trace
// 菜单：Debug → Trace → Run Trace
// 或命令：
TraceStart
// 运行一段代码
run
// 停止记录
TraceStop


// 3. 查看 Trace 记录
// 菜单：Debug → Trace → Open Run Trace
// 或命令：TraceOver（步过记录）

GDB 中的等价操作需要 rr：

# GDB + rr 版本
rr record ./target
rr replay
# 然后用 reverse-step / reverse-continue 回溯

x64dbg 的 Trace 更适合短时间的执行记录分析，不需要额外安装工具。

10. 总结

从 GDB 转到 x64dbg，核心思路的转变是：从纯命令行驱动，转向 GUI + 命令行混合驱动。x64dbg 的命令行窗口支持大部分 GDB 风格的操作（bp、run、sti、log），但 GUI 窗口的实时信息展示和右键交互能力才是效率提升的关键。

要发挥 x64dbg 100% 的能力，建议重点掌握以下几点：

  • 断点系统的全部类型：软件、硬件、内存、条件、日志断点，每种都有特定用途
  • Trace 功能：用于分析未知代码的执行路径
  • JavaScript 脚本：用于自动化重复操作（批量 Hook、参数提取、异常监控）
  • 插件生态：xAnalyzer 做代码分析、Scylla 做脱壳、SharpOD 做反反调试
  • Windows 特有机制：SEH 异常处理、PEB/TEB 结构、Wow64 兼容层

GDB 的经验不会浪费——底层的逆向思维（控制流分析、数据流追踪、调用约定理解）是通用的。x64dbg 只是把这些能力用 Windows 的方式重新组织了一遍。
