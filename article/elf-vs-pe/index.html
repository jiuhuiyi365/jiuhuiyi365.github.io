1. 为什么需要对比

理解两种格式的异同，有助于：

  • 从一个平台的逆向经验快速迁移到另一个平台
  • 理解跨平台程序（如 Chromium、Firefox）在不同系统上的行为差异
  • 编写跨平台安全工具时的设计参考
  • 理解漏洞利用技术在两个平台上的不同实现方式

设计理念差异：

ELF 的设计更简洁——一个 ELF Header + 两组表（Program Header Table + Section Header Table）就描述了所有信息。PE 的设计更复杂，因为要兼容 DOS 和 COFF 历史，文件头由多层结构嵌套组成。

ELF 的两层视图（链接视图 + 执行视图）是其独特设计。PE 没有这种区分，Section Table 同时服务于链接和加载两个目的。

ELF 中 PT_LOAD 段可以包含多个 Section（比如一个段包含 .text 和 .rodata）。PE 中每个 Section 独立映射，但 PE Loader 可能合并属性相同的 Section。这是两者在内存布局上最重要的区别。

5. 动态链接机制对比

机制ELFPE
延迟绑定PLT + GOT（Lazy Binding）IAT + PE Loader（非延迟，启动时全部解析）
延迟加载不原生支持Delay Import Directory
动态链接器ld.so（/lib64/ld-linux-x86-64.so.2）ntdll.dll（NtMapViewOfSection）
符号解析哈希表（DT_HASH / DT_GNU_HASH）按名称查找（Hint + Name 优化）
库搜索路径DT_RPATH / DT_RUNPATH / LD_LIBRARY_PATHLoadLibrary 搜索顺序
符号可见性STV_DEFAULT / STV_HIDDEN / STV_PROTECTED__declspec(dllexport) / .def 文件
弱符号STB_WEAK 支持不支持（有 __declspec(selectany) 类似）

Lazy Binding vs Eager Binding：

ELF 的 PLT/GOT 机制支持延迟绑定——函数在首次调用时才被解析。PE 默认在加载时就解析所有导入函数地址（Eager Binding），但支持 Delay Import Directory 实现类似效果。

// ELF 延迟绑定流程
call printf@plt → jmp *GOT[printf] → 首次：ld.so 解析
                                    → 非首次：直接跳转

// PE 加载时绑定
NtCreateSection → 遍历 Import Directory
               → LoadLibrary 每个 DLL
               → GetProcAddress 填充每个 IAT 条目
               → 全部完成后跳转入口点

PE 的加载时间更长（因为要解析所有导入），但运行时调用开销更小（不需要经过 PLT 跳板）。ELF 的启动更快（延迟绑定），但首次调用有额外开销。

6. 符号与导出对比

维度ELFPE
符号表位置.symtab / .dynsym 节（文件内）导出表/导入表（文件内）+ PDB（外部）
导出函数全局符号 + .dynamicExport Directory（edata）
导入函数未定义符号 + DT_NEEDEDImport Directory（idata）
查找方式哈希表（DT_GNU_HASH，O(log n)）Hint 优化的名称查找（O(n)）
按序号导出不支持支持（Base + Ordinal）
符号装饰无C++ 名称修饰（Name Mangling）
调试符号DWARF 格式（.debug_* 节）PDB 格式（外部 .pdb 文件）

ELF 把完整的符号表（包括局部变量、静态函数）放在 .symtab 中，调试信息也嵌在文件里。PE 则把详细符号信息（类型、行号、局部变量）放在外部的 PDB 文件中，文件本身只保留导出/导入表中的外部符号。

这意味着 ELF 文件的逆向信息更丰富（strip 之前），而 PE 文件的逆向高度依赖是否拥有 PDB 文件。

粒度按重定位类型区分不同计算方式按页组织，只有绝对地址修正
时机链接时（静态）+ 加载时（动态）仅加载时（ASLR 命中时）
是否必须PIE 程序必须DLL 必须，EXE 可选

ELF 的重定位更复杂但也更灵活，支持多种重定位类型（地址计算、GOT 填充、PLT 解析等）。PE 的重定位相对简单——只处理绝对地址引用，将 Delta（实际基址 - 首选基址）加到需要修正的位置上。

控制流保护-fcf-protection（CET）CFG / XFG（Guard CF 表在 .gfids 节）
代码完整性不原生支持Authenticode 数字签名
RELROPartial RELRO / Full RELRO无对应机制
FORTIFY_SOURCE编译时替换危险函数/GS + __security_check_cookie
TIP
ELF 的 RELRO（Read-Only Relocation）是一个独特的安全机制：Partial RELRO 将 .got 设为只读（加载后），Full RELRO 还会让 .got.plt 也被解析后设为只读，完全消除 GOT 覆盖攻击的可能。PE 没有类似机制。

9. 调用约定对比

调用约定决定了函数参数如何传递、栈由谁清理，这直接影响逆向分析时的参数识别。

Shadow Space 是 Windows x64 特有的概念：调用者必须在栈上预留 32 字节，供被调用者存放 RCX/RDX/R8/R9 的副本。Linux 没有这个要求。逆向时，Windows 函数的前几条指令经常会把参数保存到这个区域（mov [rsp+8], rcx），在 Linux 上看不到这种代码。

11. 逆向分析视角总结

从 GDB + ELF 逆向转向 x64dbg + PE 逆向时，需要关注的核心差异：

  • 符号信息的获取方式不同：ELF 内嵌符号表，PE 依赖 PDB。拿到一个没有 PDB 的 PE 文件，你只能看到导出表中的函数名，内部函数全是地址。ELF 即使 strip 了，导出表和 GOT/PLT 的符号名仍然保留。
  • 动态链接分析思路不同：ELF 关注 PLT/GOT 的延迟绑定行为，PE 关注 IAT 的静态绑定。分析 ELF 需要理解 PLT stub 的跳转逻辑，分析 PE 只需要看 IAT 表被填了什么地址。
  • 调用约定不同：识别函数参数时，Linux x64 看 RDI/RSI/RDX/RCX，Windows x64 看 RCX/RDX/R8/R9。Shadow Space 的存在让 Windows 函数的 prologue 更长。
  • 安全机制不同：ELF 的 RELRO 保护 GOT，PE 没有类似保护（但有 CFG 保护间接调用）。漏洞利用时，ELF 可以尝试 GOT 覆盖，PE 则需要找 ROP 链或利用 CFG 绕过。
  • 调试工具不同：ELF 用 GDB + readelf + objdump，PE 用 x64dbg + PE-bear + dumpbin。思维模式是一样的，只是命令和工具换了。
